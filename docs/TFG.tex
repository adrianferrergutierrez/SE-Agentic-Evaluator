\chapter*{Resum}
\label{chap:resum}
\addcontentsline{toc}{chapter}{Resum}

Els sistemes actuals d'avaluaci\'o autom\`atica de treballs acad\`emics solen funcionar com a pipelines r\'igids: sempre segueixen el mateix ordre de passos, no poden analitzar diagrames ni figures t\`ecniques, i obliguen a modificar el codi font cada vegada que canvia la r\'ubrica d'avaluaci\'o. \'Es el cas de SE-rubric-evaluAItor, una eina desenvolupada dins l'ecosistema educatiu LAMB que, tot i ser funcional, resulta poc flexible i dif\'icil d'adaptar a nous tipus d'entregables.

Aquest Treball de Fi de Grau presenta \textbf{Evaluaitor-Lamb}, un sistema que substitueix aquest pipeline r\'igid per un agent d'intel\textperiodcentered lig\`encia artificial capa\c{c} de decidir per si mateix quins passos seguir per avaluar cada document, adaptant-se al seu contingut concret. El sistema s'ha constru\"it des de zero, sense dependre de cap plataforma externa, seguint la metodologia ``Agents All the Way Down'' \cite{alier2025} del professor Marc Alier i altres.

Per fer-ho possible, el sistema combina tres idees clau. Primer, disposa d'un conjunt de 18 eines especialitzades (extracci\'o de text, an\`alisi de diagrames, generaci\'o d'informes, etc.) que l'agent invoca segons les necessitats de cada cas. Segon, genera autom\`aticament un pla de treball personalitzat per a cada r\'ubrica i el reutilitza amb m\'ultiples documents, estalviant temps i costos. I tercer, delega sempre el c\`alcul de les notes a codi Python ---mai al model d'IA--- de manera que cada puntuaci\'o quedi vinculada a una evid\`encia concreta del document, evitant errors num\`erics. El sistema tamb\'e interpreta diagrames i imatges t\`ecniques, i pot funcionar amb diversos prove\"idors de models d'IA (locals o al n\'uvol) sense necessitat de tocar el codi.

Evaluaitor-Lamb s'ha validat mitjan\c{c}ant sis escenaris t\`ecnics, tots completats amb \`exit, i mitjan\c{c}ant una comparaci\'o amb una avaluadora humana experta sobre mem\`ories reals de m\`aster. La validaci\'o t\`ecnica del sistema \'es robusta; no obstant aix\`o, la comparaci\'o amb l'experta revela un biaix sistem\`atic cap a la infra-avaluaci\'o i una correlaci\'o encara feble, la qual cosa marca el cam\'i per a la futura calibraci\'o del sistema.

\textbf{Paraules clau:} agent d'IA, avaluaci\'o assistida, workflow reutilitzable, visi\'o multimodal, c\`alcul determinista, multi-prove\"idor.


\chapter*{Abstract}
\label{chap:abstract}
\addcontentsline{toc}{chapter}{Abstract}

Current automated systems for evaluating academic assignments tend to work as rigid pipelines: they always follow the same fixed sequence of steps, cannot analyze diagrams or technical figures, and require changing the source code every time the evaluation rubric changes. This is the case of SE-rubric-evaluAItor, a tool developed within the LAMB educational ecosystem that, while functional, is inflexible and hard to adapt to new types of deliverables.

This Bachelor's Thesis presents \textbf{Evaluaitor-Lamb}, a system that replaces this rigid pipeline with an AI agent capable of deciding for itself which steps to follow when evaluating each document, adapting to its specific content. The system was built from scratch, without depending on any external platform, following the ``Agents All the Way Down'' methodology \cite{alier2025} proposed by Professor Marc Alier and others.

To achieve this, the system combines three key ideas. First, it relies on a set of 18 specialized tools (text extraction, diagram analysis, report generation, etc.) that the agent invokes according to the needs of each case. Second, it automatically generates a customized workflow for each rubric and reuses it across multiple documents, saving time and cost. Third, grade calculation is always delegated to Python code ---never to the AI model--- so that every score is tied to concrete evidence from the document, avoiding numerical errors. The system can also interpret diagrams and technical images, and works with several AI model providers (local or cloud-based) without requiring any code changes.

Evaluaitor-Lamb has been validated through six technical scenarios, all successfully completed, and through a comparison against an expert human evaluator on real master's theses. The technical validation of the system is robust; however, the comparison with the human expert reveals a systematic bias towards under-evaluation and a still weak correlation, which paves the way for future system calibration.

\textbf{Keywords:} AI agent, assisted evaluation, reusable workflow, multimodal vision, deterministic grading, multi-provider.

\chapter{Introducci\'o i context}
\label{chap:introduccio}

El projecte Evaluaitor-Lamb correspon al treball de fi de grau en l'especialitat d'enginyeria del software dins el grau en Enginyeria Inform\`atica de la Facultat d'Inform\`atica de Barcelona (FIB), a la Universitat Polit\`ecnica de Catalunya (UPC).

Aquest treball de fi de grau es desenvolupa en el marc d'una beca d'aprenentatge a l'ICE. El projecte s'integra dins de l'ecosistema LAMB, una plataforma educativa que utilitza Intel·lig\`encia Artificial per assistir en processos d'aprenentatge.

Actualment, l'ecosistema LAMB compta amb diversos projectes relacionats amb l'avaluaci\'o assistida per IA, incloent-hi \href{https://github.com/Lamb-Project/SE-rubric-evaluAItor}{SE-rubric-evaluAItor}, un pipeline d'investigaci\'o per a l'avaluaci\'o de documents de requisits d'Enginyeria del Software, i \href{https://github.com/Lamb-Project/DiagramLens}{DiagramLens}, un m\`odul de visi\'o multimodal per a la descripci\'o de diagrames t\`ecnics.

L'objectiu principal d'aquest TFG \'es el disseny i implementaci\'o de Evaluaitor-Lamb, un \textbf{sistema meta-ag\`entic complet} que transforma el pipeline r\'igid de SE-rubric-evaluAItor en un assistent din\`amic, contextual i adaptatiu per a l'avaluaci\'o d'entregables acad\`emics de qualsevol tipus. El sistema implementa un \textbf{agent loop propi} seguint la metodologia ``Agents All the Way Down'' del professor Marc Alier. S’ha dissenyat amb un enfocament local-first per funcionar amb models open-source locals, i durant la fase de validació s’han utilitzat models d'IA de m\'ultiples prove\"idors cloud (DashScope, OpenRouter, Google Gemini) per a tasques d'extracci\'o i avaluaci\'o, i delega el c\`alcul de notes en scripts Python determin\'istics per evitar al·lucinacions num\`eriques dels LLMs. L'\'us de models cloud (DashScope o OpenRouter) durant el projecte \'es una elecci\'o de conveni\`encia t\`ecnica durant la fase de validaci\'o (per no disposar d'una GPU dedicada de gamma alta a l'ordinador port\`atil de l'autor). El sistema est\`a a punt per passar a producci\'o amb models locals open-source nom\'es canviant dues variables d'entorn.

El sistema disposa de \textbf{18 tools especialitzades} (extracci\'o, an\`alisi, visi\'o multimodal, grading, configuraci\'o, workflow) i implementa un \textbf{workflow generator} que genera workflows JSON personalitzats per a cada r\'ubrica, permetent la seva reutilitzaci\'o amb m\'ultiples documents. Aquesta arquitectura permet separar la generaci\'o de workflows de la seva execuci\'o, optimitzant costos i temps de processament.

\section{Definicions importants}
\label{sec:definicions}

Per a la correcta comprensi\'o de l'abast d'aquest projecte, es defineixen els seg\"uents conceptes clau del domini:

\subsection{Agent d'IA}
\label{subsec:agent-ia}

Un agent d'IA \'es un sistema de programari capa\c{c} de percebre el seu entorn, prendre decisions aut\`onomes i executar accions per assolir un objectiu definit. A difer\`encia d'un pipeline seq\"uencial predefinit, un agent pot seleccionar din\`amicament quines eines invocar i en quin ordre, adaptant-se al contingut espec\'ific de cada document. En aquest projecte, l'agent opera dins del sistema propi Evaluaitor-Lamb i disposa d'un conjunt de tools (extracci\'o, an\`alisi, visi\'o, grading) que invoca segons les necessitats detectades.

\subsection{LMS (Learning Management System)}
\label{subsec:lms}

Un LMS (Learning Management System) \'es una plataforma tecnol\`ogica o aplicaci\'o web dissenyada per crear, gestionar, distribuir i avaluar cursos en l\'inia i programes de formaci\'o de forma centralitzada. Funciona com a aula virtual, facilitant el seguiment dels alumnes, la gesti\'o de materials i la interacci\'o entre usuaris. Per exemple Moodle.

\subsection{Skill}
\label{subsec:skill}

Una ``skill'' (habilitat) a IA \'es un manual d'instruccions reutilitzable, estructurat en format Markdown, que ensenya a un agent d'intel·lig\`encia artificial (com Claude, ChatGPT o Gemini) com executar una tasca espec\'ifica. Permet encapsular el pas a pas, exemples i regles per evitar haver de repetir prompts llargs constantment. Les Skills permeten estendre les capacitats d'un agent d'IA per a dominis espec\'ifics sense modificar el nucli del sistema.

\subsection{Agent Loop}
\label{subsec:agent-loop}

Un Agent Loop \'es el cicle fonamental d'execuci\'o d'un agent d'IA. Consisteix en un bucle que: (1) envia el context actual al LLM, (2) rep la resposta del LLM, (3) si la resposta inclou tool calls, executa les tools i afegeix els resultats al context, (4) repeteix el proc\'es fins que el LLM no solicita m\'es tool calls. En aquest projecte, l'agent loop s'implementa en un codi Python, seguint la metodologia ``Agents All the Way Down'' del professor Marc Alier, sense dependre de cap framework extern.

\subsection{LLM (Large Language Model)}
\label{subsec:llm}

Un Large Language Model (LLM) \'es un model d'intel·lig\`encia artificial entrenat amb grans volums de text capa\c{c} de generar, comprendre i transformar llenguatge natural. En aquest projecte s'utilitzen LLMs de diversos prove\"idors (DashScope/Alibaba Cloud, OpenRouter, Google Gemini) per a tasques d'extracci\'o d'informaci\'o, an\`alisi de qualitat i avaluaci\'o de documents. El sistema implementa una capa d'abstracci\'o que permet intercanviar prove\"idors sense modificar la l\`ogica d'avaluaci\'o. L’arquitectura està preparada per treballar preferentment amb models locals open-source, tot i que durant la validació s’han fet servir proveïdors cloud.

\subsection{DashScope}

DashScope \'es la plataforma de serveis d'IA d'Alibaba Cloud que proporciona acc\'es a models de llenguatge i visi\'o de la fam\'ilia Qwen. A difer\`encia d'Ollama (execuci\'o local), DashScope opera com a servei cloud, oferint models de major qualitat sense requerir hardware especialitzat. En aquest projecte, DashScope s’ha utilitzat com a entorn de prototipat i validació per la disponibilitat immediata de recursos, mentre que el disseny del sistema manté una orientació local-first, per\`o el sistema suporta m\'ultiples prove\"idors (OpenRouter, Google Gemini, Ollama local) mitjan\c{c}ant una capa d'abstracci\'o configurada via variables d'entorn (Secci\'o~\ref{subsec:multi-provider}).

\subsection{Workflow Generator/Executor}
\label{subsec:workflow}

El Workflow Generator \'es un component que genera workflows JSON personalitzats per a cada r\'ubrica d'avaluaci\'o. Analitza la r\'ubrica i determina quines tools s'han d'executar i en quin ordre. El Workflow Executor \'es el component que executa aquests workflows amb documents espec\'ifics. Aquesta separaci\'o permet generar un workflow una vegada i reutilitzar-lo amb m\'ultiples documents, optimitzant costos i temps de processament.

\subsection{Tool Registry}
\label{subsec:tool-registry}

El Tool Registry \'es un registre centralitzat de totes les tools disponibles per a l'agent. Cada tool t\'e un nom, descripci\'o, categoria i funci\'o d'execuci\'o. En aquest projecte, el registry cont\'e 18 tools especialitzades en extracci\'o (docx\_extract, xlsx\_to\_markdown, describe\_diagrams, read\_file, list\_directory), an\`alisi (extract\_objectives, detect\_orphans, evaluate\_smart), avaluaci\'o (criterion\_evaluator, grader, report\_generator) i workflow (generate\_workflow, execute\_workflow). L'agent consulta el registry per determinar quines tools pot utilitzar.

\subsection{R\'ubrica d'Avaluaci\'o amb IA}
\label{subsec:rubrica-ia}

Estrat\`egia mitjan\c{c}ant la qual s'instrueix un model de llenguatge (LLM) per avaluar un document acad\`emic basant-se en criteris predefinits (objectius, requisits, casos d'\'us, tra\c{c}abilitat), amb pesos configurables per a cada criteri. La nota final es calcula de manera determin\'istica amb Python, no pel LLM.

\subsection{Visi\'o Multimodal}
\label{subsec:visio-multimodal}

Capacitat d'un model d'IA de processar i comprendre tant text com imatges. En aquest projecte, el model de visi\'o (configurable via \texttt{DASHSCOPE\_VISION\_MODEL}) s'utilitza per categoritzar i descriure t\`ecnicament diagrames UML i figures t\`ecniques embebudes en documents Markdown, proporcionant evid\`encia visual per a l'avaluaci\'o de consist\`encia arquitect\`onica.

\subsection{RGPD (Reglament General de Protecci\'o de Dades)}
\label{subsec:rgpd}

Reglament europeu que estableix el marc legal per a la protecci\'o de dades personals. En el context d'aquest projecte, l'objectiu central de disseny \'es garantir la confidencialitat absoluta de les dades acad\`emiques mitjan\c{c}ant l'execuci\'o de l'agent en local amb models Open Source (lliures i de codi obert, com la fam\'ilia Qwen o Llama via Ollama). Aix\`o assegura que cap dada dels estudiants surti de la infraestructura de la universitat, complint estrictament amb el RGPD. Opcionalment, per a entorns de desenvolupament o proves, el suport multi-provider permet connectar amb prove\"idors cloud que compleixen amb les normatives europees (com DashScope/Alibaba Cloud amb centres de dades a Europa).

\section{Identificaci\'o del problema}
\label{sec:identificacio-problema}

El problema principal que motiva aquest projecte \'es la rigidesa i manca d'adaptabilitat del sistema actual d'avaluaci\'o de l'ecosistema LAMB, aix\'i com les limitacions de les plataformes ag\`entiques existents per a casos d'\'us espec\'ifics d'avaluaci\'o acad\`emica.

Actualment, el projecte \href{https://github.com/Lamb-Project/SE-rubric-evaluAItor}{SE-rubric-evaluAItor} implementa un pipeline seq\"uencial d'avaluaci\'o amb les seg\"uents limitacions:

\begin{enumerate}
    \item \textbf{Flux R\'igid:} El pipeline executa les fases en un ordre fix predefinit (extracci\'o $\rightarrow$ an\`alisi $\rightarrow$ avaluaci\'o $\rightarrow$ grading), sense capacitat d'adaptar-se al contingut espec\'ific de cada document.
    \item \textbf{Configuraci\'o Hardcodejada:} Els pesos de la r\'ubrica estan fixats al codi font (\texttt{grader.py}). Canviar la r\'ubrica per a un altre tipus d'entregable requereix modificar i redeplegar el codi.
    \item \textbf{Sense Visi\'o Multimodal:} El sistema no pot analitzar diagrames UML ni figures t\`ecniques embebudes, elements clau en entregables d'Enginyeria del Software.
    \item \textbf{Manca de Tra\c{c}abilitat d'Evid\`encia:} Els judicis evaluatius no estan sempre vinculats a evid\`encia textual o visual expl\'icita del document, dificultant l'auditoria i la transpar\`encia.
    \item \textbf{Sense Processament per Lots:} El sistema avalua un document a la vegada, fent-lo poc pr\`actic per a professors amb desenes d'alumnes.
    \item \textbf{Manca de Workflows Reutilitzables:} No existeix un mecanisme per generar workflows personalitzats per a cada r\'ubrica i reutilitzar-los amb m\'ultiples documents, obligant a repetir el mateix proc\'es de configuraci\'o per a cada avaluaci\'o.
\end{enumerate}

Addicionalment, les plataformes ag\`entiques existents (com opencode.ai) presenten limitacions per a aquest cas d'\'us espec\'ific:

\begin{enumerate}[start=7]
    \item \textbf{Depend\`encia de Plataforma:} Les Skills estan acoblades a la plataforma espec\'ifica, limitant la portabilitat i reutilitzaci\'o en altres contextos.
    \item \textbf{Manca de Control sobre el Flux:} La plataforma gestiona internament el loop de l'agent, impedint implementar l\`ogica personalitzada de gesti\'o de workflows i sessions.
    \item \textbf{Limitacions de Models Locals:} Els models locals (Ollama) presenten limitacions en qualitat d'extracci\'o estructurada i raonament avaluatiu per a tasques complexes.
\end{enumerate}

La soluci\'o proposada, Evaluaitor-Lamb, resol totes aquestes limitacions implementant un \textbf{sistema meta-ag\`entic complet} amb agent loop propi, workflow generator/executor, 18 tools especialitzades, visi\'o multimodal multi-provider, i seguretat al scaffolding.

\section{Evoluci\'o del projecte}
\label{sec:evolucio}

El projecte actual, Evaluaitor-Lamb, representa una evoluci\'o significativa respecte a la planificaci\'o inicial documentada al Lliurable 1 d'aquest TFG. Aquesta secci\'o explica els motius i l'abast dels canvis.

\subsection{Context inicial}

La proposta original del TFG, titulada ``Evaluaitor LAMB'', consistia en el disseny i implementaci\'o d'un \textbf{microservei dedicat} a l'avaluaci\'o autom\`atica de tasques educatives mitjan\c{c}ant IA. L'objectiu era desacoblar la l\`ogica d'avaluaci\'o de la capa d'integraci\'o amb l'LMS (LAMBA), seguint una arquitectura de microserveis amb cues as\'incrones, sistema de plugins i multi-tenan\c{c}a.

\subsection{Motius del canvi}

Durant les primeres fases de desenvolupament, es va fer evident que l'ecosistema LAMB \'es un entorn \textbf{altament din\`amic i en constant evoluci\'o}. Els canvis freq\"uents en l'arquitectura, les depend\`encies i les prioritats del projecte LAMB feien que la planificaci\'o original d'un microservei acoblat a l'ecosistema fos dif\'icil de mantenir alineada amb la realitat del projecte.

Concretament:

\begin{itemize}
    \item L'arquitectura de LAMB estava experimentant canvis significatius que afectaven directament la integraci\'o prevista amb LAMBA i lamb-kb-server.
    \item La necessitat de desacoblar l'avaluaci\'o de la capa LTI seguia sent v\`alida, per\`o l'enfocament de microservei requeria una infraestructura i coordinaci\'o que depassaven l'abast d'un TFG individual.
    \item Simult\`aniament, l'equip LAMB estava explorant noves aproximacions basades en \textbf{agents d'IA} i la plataforma \textbf{opencode.ai}, que oferien una via m\'es flexible i menys acoblada per abordar el mateix problema de fons: millorar l'avaluaci\'o assistida per IA.
\end{itemize}

\subsection{Decisi\'o de reorientaci\'o}

Davant d'aquest context, es va prendre la decisi\'o conjuntament amb el director del TFG de \textbf{reorientar el projecte} cap a una \textbf{Skill ag\`entica} per a opencode.ai. Aquesta nova direcci\'o:

\begin{itemize}
    \item \textbf{Mant\'e l'objectiu central:} avaluar documents acad\`emics amb IA.
    \item \textbf{Redueix l'acoblament:} la Skill opera de manera independent de l'arquitectura espec\'ifica de LAMB.
    \item \textbf{Afegeix valor diferencial:} integra visi\'o multimodal local, evid\`encia obligat\`oria i c\`alcul determin\'istic de notes.
    \item \textbf{\'Es m\'es resilient als canvis:} en no dependre d'una arquitectura de microserveis espec\'ifica, el projecte \'es menys vulnerable als canvis constants de LAMB.
\end{itemize}

\subsection{Segona reorientaci\'o: De Skill ag\`entica a sistema meta-ag\`entic (Mar\c{c}-Juny 2026)}

\subsubsection{Context de la segona evoluci\'o}

Durant el desenvolupament de la Skill ag\`entica per a opencode.ai, es van identificar diverses limitacions que motivaven una nova evoluci\'o del projecte:

\begin{enumerate}
    \item \textbf{Depend\`encia de plataforma:} La Skill estava acoblada a opencode.ai, limitant la seva portabilitat i reutilitzaci\'o en altres contextos.
    \item \textbf{Manca de control sobre el flux d'execuci\'o:} La plataforma opencode.ai gestionava internament el loop de l'agent, impedint implementar l\`ogica personalitzada de gesti\'o de workflows i sessions.
    \item \textbf{Limitacions de models locals:} Els models locals via Ollama (qwen3:14b, qwen3-vl:7b) presentaven limitacions en qualitat d'extracci\'o estructurada i raonament avaluatiu, especialment per a tasques complexes com l'an\`alisi de diagrames UML.
    \item \textbf{Necessitat de workflows reutilitzables:} Es va identificar la necessitat de separar la generaci\'o de workflows de la seva execuci\'o, permetent crear workflows personalitzats per a cada r\'ubrica i reutilitzar-los amb m\'ultiples documents.
\end{enumerate}

\subsubsection{Decisi\'o de segona reorientaci\'o}

Davant d'aquest context, es va prendre la decisi\'o conjuntament amb el director del TFG de \textbf{evolucionar el projecte} cap a un \textbf{sistema meta-ag\`entic complet} amb les seg\"uents caracter\'istiques:

\begin{itemize}
    \item \textbf{Agent loop propi:} Implementaci\'o d'un agent loop independent que no dep\`en de cap plataforma externa, seguint la metodologia ``Agents All the Way Down'' del professor Marc Alier.
    \item \textbf{Canvi a DashScope i suport multi-provider:} Migraci\'o d'Ollama a DashScope (Alibaba Cloud) com a prove\"idor primari per a la validaci\'o del sistema degut a limitacions de maquinari, dissenyant l'arquitectura amb suport multi-prove\"idor (OpenRouter, Google Gemini i execuci\'o en local mantinguda) configurable mitjan\c{c}ant variables d'entorn, millorant significativament la qualitat d'extracci\'o, l'avaluaci\'o i la resili\`encia del sistema.
    \item \textbf{Workflow generator i executor:} Implementaci\'o de dos components independents:
    \begin{itemize}
        \item \texttt{generate\_workflow}: Genera workflows JSON personalitzats per a cada r\'ubrica.
        \item \texttt{execute\_workflow}: Executa workflows existents amb documents espec\'ifics.
    \end{itemize}
    \item \textbf{18 tools especialitzades:} Registry complet amb tools d'extracci\'o, an\`alisi, avaluaci\'o, visi\'o, configuraci\'o i workflow.
    \item \textbf{Seguretat al scaffolding:} Implementaci\'o de security policy amb allow-list al scaffolding (no al prompt), seguint les bones pr\`actiques de la metodologia.
    \item \textbf{Validaci\'o amb escenaris reals:} Disseny de 6 escenaris de validaci\'o completats amb \`exit:
    \begin{itemize}
        \item Escenari 1: Avaluaci\'o completa
        \item Escenari 2: Nom\'es generar workflow
        \item Escenari 3: Executar workflow existent
        \item Escenari 4: Continuaci\'o de sessi\'o
        \item Escenari 5: Input adversarial
        \item Escenari 6: Workflow amb visi\'o multimodal
    \end{itemize}
\end{itemize}

\subsubsection{Impacte d'aquesta segona evoluci\'o}

Aquesta segona reorientaci\'o representa una \textbf{millora significativa} respecte a la versi\'o anterior:

\begin{itemize}
    \item \textbf{Major flexibilitat:} El sistema ja no dep\`en d'opencode.ai i pot executar-se en qualsevol entorn Python.
    \item \textbf{Millor qualitat:} El pas d'una soluci\'o exclusivament local a una arquitectura multi-prove\"idor permet aprofitar la superioritat dels models cloud (DashScope, OpenRouter, Gemini) en qualitat d'extracci\'o i avaluaci\'o, oferint ara la llibertat d'alternar indistintament entre l'execuci\'o en local o en el cloud segons les necessitats, a m\'es de garantir disponibilitat cont\'inua.
    \item \textbf{Reutilitzaci\'o de workflows:} La separaci\'o entre generaci\'o i execuci\'o permet optimitzar costos i temps.
    \item \textbf{Validaci\'o rigorosa:} Els 6 escenaris validats demostren la robustesa del sistema en casos d'\'us reals.
\end{itemize}

Aquesta evoluci\'o es documenta detalladament a les seccions d'\textbf{Implementaci\'o} i \textbf{Validaci\'o} d'aquesta mem\`oria.

\chapter{Estat de l'art i treballs relacionats}
\label{chap:estat-art}

Aquest cap\'itol revisa l'estat de l'art en les tres \`aries tem\`atiques fonamentals del projecte: l'avaluaci\'o autom\`atica amb IA, els frameworks i metodologies ag\`entiques, i la visi\'o multimodal per a documents t\`ecnics. L'objectiu \'es contextualitzar Evaluaitor-Lamb dins del paisatge cient\'ific i tecnol\`ogic actual, identificant les mancances que el projecte aborda.

\section{Avaluaci\'o autom\`atica assistida per IA}
\label{sec:aval-auto-ia}

\subsection{Automated Essay Scoring (AES)}
\label{subsec:aes}

L'avaluaci\'o autom\`atica d'escrits (AES, \textit{Automated Essay Scoring}) \'es un camp d'investigaci\'o amb m\'es de dues d\`ecades d'hist\`oria. Els primers sistemes, com \textbf{e-rater} \cite{burstein2013}, desenvolupat per ETS (Educational Testing Service), utilitzaven caracter\'istiques ling\"u\'istiques extr\'iques (longitud d'oracions, diversitat l\`exica, errors gramaticals) per predir puntuacions, sense comprendre realment el contingut sem\`antic del text. Aquest enfocament presentava limitacions evidents: un escrit que repet\'is paraules clau sense coher\`encia podia obtenir puntuacions altes \cite{perelman2014}.

La segona generaci\'o d'eines, com \textbf{CRAS} (\textit{Coh-Metrix Research Assessment System}) \cite{graesser2014}, va integrar m\`etriques de cohesi\'o textual i coher\`encia narrativa, per\`o continuava depenent de caracter\'istiques extr\'iques predissenyades, sense capacitat d'adaptaci\'o a dominis espec\'ifics.

La tercera i actual generaci\'o aprofita els \textbf{Large Language Models (LLMs)}. Treballs recents demostren que models com GPT-4 poden aproximar les puntuacions d'avaluadors humans amb una correlaci\'o de Pearson $r > 0{,}80$ en dominis generals \cite{guo2024}. No obstant aix\`o, aquests resultats amaguen problemes fonamentals:

\begin{enumerate}
    \item \textbf{Al·lucinacions num\`eriques:} Els LLMs generen puntuacions que no es corresponen amb el raonament textual que els acompanya, produint inconsist\`encies entre l'explicaci\'o i la nota final \cite{guo2024}.
    \item \textbf{Manca de transpar\`encia:} La majoria de sistemes no vinculen cada judici evaluatiu a evid\`encia textual concreta del document avaluat, dificultant l'auditoria i la confian\c{c}a del docent.
    \item \textbf{Biaix de longitud:} Els LLMs tendeixen a correlacionar positivament la longitud del text amb la qualitat, independentment del contingut real \cite{perelman2014}.
    \item \textbf{Sensibilitat al prompt:} Petites variacions en la instrucci\'o d'avaluaci\'o produeixen canvis significatius en les puntuacions \cite{guo2024}.
\end{enumerate}

Evaluaitor-Lamb aborda directament els problemes 1 i 2 mitjan\c{c}ant dos mecanismes: (a) el c\`alcul determin\'istic de notes amb scripts Python, eliminant les al·lucinacions num\`eriques, i (b) la tra\c{c}abilitat obligat\`oria d'evid\`encia, que exigeix que cada puntuaci\'o estigui vinculada a una cita textual o descripci\'o de diagrama del document.

\subsection{LLM-as-a-Judge}
\label{subsec:llm-judge}

El paradigma \textit{LLM-as-a-Judge} \cite{zheng2023} utilitza un model de llenguatge com a avaluador de sortides generades per un altre model. S'ha popularitzat en l'avaluaci\'o de sistemes de di\`aleg i generaci\'o de codi, per\`o la seva aplicaci\'o directa a l'avaluaci\'o d'entregables acad\`emics presenta limitacions espec\'ifiques:

\begin{itemize}
    \item \textbf{Autoconsist\`encia limitada:} El mateix model pot produir puntuacions diferents per al mateix document en execucions successives \cite{zheng2023}.
    \item \textbf{Biaix de posicionament:} En avaluacions per parelles, el model tendeix a preferir la primera opci\'o presentada \cite{zheng2023}.
    \item \textbf{Manca de replicabilitat:} Sense un workflow determin\'istic, dos usuaris poden obtenir resultats diferents amb el mateix model i document.
\end{itemize}

Evaluaitor-Lamb mitiga aquests problemes mitjan\c{c}ant la separaci\'o entre la generaci\'o de judicis qualitatius (que delega al LLM) i l'agregaci\'o num\`erica (que realitza un script determin\'istic), garantint que la nota final sigui 100\% replicable donades les mateixes puntuacions parcials.

\subsection{Avaluaci\'o de documents d'Enginyeria del Software}
\label{subsec:aval-es}

L'avaluaci\'o d'entregables d'Enginyeria del Software difereix significativament de l'avaluaci\'o d'escrits generals per diverses raons:

\begin{enumerate}
    \item \textbf{Estructura sem\`antica:} Els documents t\`ecnics contenen elements estructurals amb sem\`antica pr\`opia (objectius, requisits, casos d'\'us, diagrames UML) que no es poden avaluar com a text pla.
    \item \textbf{Tra\c{c}abilitat:} La qualitat d'un document de requisits dep\`en de les relacions entre elements (p.ex., cada requisit ha d'estar vinculat a un objectiu), no nom\'es del text a\"illat.
    \item \textbf{Component visual:} Els diagrames UML i les figures t\`ecniques s\'on part integral del document i la seva avaluaci\'o requereix comprensi\'o visual, no nom\'es textual.
\end{enumerate}

El projecte \textbf{SE-rubric-evaluAItor} \cite{se-rubric} va ser un primer intent d'abordar aquestes especificitats amb un pipeline seq\"uencial, per\`o les seves limitacions (flux r\'igid, r\'ubriques hardcodejades, sense visi\'o) van motivar el desenvolupament de Evaluaitor-Lamb.

\section{Frameworks i metodologies ag\`entiques}
\label{sec:frameworks-agentics}

\subsection{Frameworks d'agents existents}
\label{subsec:frameworks}

L'ecosistema de frameworks per a sistemes ag\`entics ha experimentat un creixement exponencial des del 2023. Els m\'es rellevants s\'on:

\textbf{LangChain / LangGraph} \cite{langchain2024}: Framework de refer\`encia per a la construcci\'o d'aplicacions amb LLMs. Proporciona abstraccions per a cadenes de crides, gesti\'o de mem\`oria, i integraci\'o amb fonts de dades externes. LangGraph, la seva extensi\'o per a agents amb estats, permet definir grafs d'execuci\'o amb cicles i bifurcacions. Limitaci\'o principal: l'alta abstracci\'o dificulta el control fi sobre el proc\'es d'avaluaci\'o i la gesti\'o de sessions.

\textbf{AutoGen} \cite{wu2023}: Framework de Microsoft per a converses multi-agent. Permet definir agents amb rols especialitzats que col·laboren mitjan\c{c}ant converses estructurades. Limitaci\'o: el model de conversa entre agents \'es massa gen\`eric per a l'avaluaci\'o estructurada amb r\'ubriques, on el flux d'execuci\'o est\`a determinat pels criteris i no per la negociaci\'o entre agents.

\textbf{CrewAI} \cite{crewai2024}: Framework per a orquestraci\'o d'equips d'agents amb rols, objectius i tasques definides. Limitaci\'o: enfocament en productivitat gen\`erica, sense mecanismes natius per a avaluaci\'o amb r\'ubriques, tra\c{c}abilitat d'evid\`encia o c\`alcul determin\'istic de notes.

\textbf{Claude Code / opencode.ai} \cite{opencode}: Plataformes que implementen agents de codi amb acc\'es a terminal, fitxers i eines. Proporcionen un agent loop gestit per la plataforma amb un sistema de Skills (instruccions en Markdown). Limitaci\'o: l'agent loop \'es propietat de la plataforma, impedint la implementaci\'o de l\`ogica personalitzada de workflows i sessions, i creant depend\`encia de plataforma.

Cap d'aquests frameworks proporciona nativament els mecanismes que Evaluaitor-Lamb requereix: r\'ubriques configurables externament, tra\c{c}abilitat d'evid\`encia obligat\`oria, c\`alcul determin\'istic de notes, visi\'o multimodal per a diagrames UML, i workflows reutilitzables. Aquesta mancan\`encia fonamental justifica la decisi\'o d'implementar un agent loop propi.

\subsection{La metodologia ``Agents All the Way Down''}
\label{subsec:aawd}

La metodologia \textbf{``Agents All the Way Down''} \cite{alier2025}, desenvolupada pel professor Marc Alier de la UPC entre d'altres, estableix un marc conceptual per a la construcci\'o de sistemes meta-ag\`entics fiables. La metodologia defineix cinc fases de maduresa (P1--P5):

\begin{enumerate}
    \item \textbf{P1 (Substrate):} Internalitzar el LLM com a component de software, comprendre la disciplina KV-cache (tools i system est\`atics, messages en append-only) per optimitzar l'\'us de la mem\`oria de context.
    \item \textbf{P2 (Building Blocks):} Implementar els blocs fonamentals: function calling amb esquema JSON, tool registry tipus, security policy al scaffolding (allow-list, no al prompt), i session management per a converses multi-turn.
    \item \textbf{P3 (Prototype):} Prototipar amb un agent de prop\`osit general (p.ex., Claude Code) per validar l'arquitectura emergent i refinar els prompts de les tools.
    \item \textbf{P4 (Ship as CLI):} Empaquetar l'agent com a CLI seguint el \textit{Turtle Pattern} --- un agent nadiu de terminal que pot ser invocat i compost des de la l\'inia de comandes, permetent l'orquestraci\'o d'agents.
    \item \textbf{P5 (Agent-Tests-Agent):} Implementar behavioral testing amb un agent avaluador que condueixi l'agent principal amb escenaris de prova automatitzats.
\end{enumerate}

Aquesta metodologia \'es particularment rellevant per a Evaluaitor-Lamb perqu\`e estableix principis de disseny que resolen problemes pr\`actics dels frameworks existents: la seguretat al scaffolding (P2) evita atacs de prompt injection, la disciplina KV-cache (P1) optimitza el consum de tokens, i l'empaquetat com a CLI (P4) garanteix la portabilitat i composabilitat del sistema.

\section{Models de llenguatge per a avaluaci\'o}
\label{sec:models-ll}

\subsection{La fam\'ilia Qwen i DashScope}
\label{subsec:qwen}

Els models de la fam\'ilia \textbf{Qwen} \cite{qwen2024}, desenvolupats per Alibaba Cloud, representen una de les fam\'ilies de LLMs de codi obert amb millor rendiment en tasques de comprensi\'o i generaci\'o en m\'ultiples idiomes, incloent-hi el catal\`a i el castell\`a. La plataforma \textbf{DashScope} proporciona acc\'es cloud a aquests models, oferint variants optimitzades per a diferents tasques:

\begin{itemize}
    \item \textbf{Qwen3.6-plus:} Model de text de alta qualitat per a tasques d'extracci\'o estructurada, an\`alisi de qualitat i avaluaci\'o de criteris.
    \item \textbf{Qwen-VL-Max:} Model de visi\'o multimodal per a descripci\'o i categoritzaci\'o de diagrames UML i figures t\`ecniques.
\end{itemize}

L'elecci\'o de DashScope sobre altres prove\"idors (OpenAI, Anthropic) es basa en tres factors: (1) compliment RGPD mitjan\c{c}ant centres de dades a Europa, (2) qualitat competitiva en llenguatges no anglesos, i (3) model de preus per \'us que permet controlar costos.

No obstant aix\`o, durant el desenvolupament del projecte es va identificar la conveni\`encia de dissenyar una arquitectura m\'ulti-prove\"idor, no nom\'es per mitigar les limitacions de quota d'un \'unic servei cloud, sin\'o tamb\'e per oferir una flexibilitat total entre l'entorn local i el nuvol. Evaluaitor-Lamb ara suporta la configuraci\'o indistinta de prove\"idors alternatius (OpenRouter, Google Gemini o models locals via Ollama) mitjan\c{c}ant variables d'entorn, garantint el desacoblement entre la l\`ogica d'avaluaci\'o i el prove\"idor de models i permetent l'execuci\'o en local de manera immediata.

\subsection{Models de visi\'o multimodal}
\label{subsec:models-visio}

La capacitat de processar imatges juntament amb text \'es fonamental per a l'avaluaci\'o de documents. Trobem un exemple clar en l'Enginyeria, on els diagrames UML s\'on components essencials. Els models de visi\'o multimodal han evolucionat r\`apidament:

\begin{itemize}
    \item \textbf{GPT-4V} \cite{openai2024}: Model de refer\`encia en visi\'o, per\`o amb cost elevat i sense compliment RGPD directe.
    \item \textbf{Qwen-VL-Max} \cite{qwen2024}: Alternativa competitiva amb millor suport multiling\"ue i compliment RGPD.
    \item \textbf{Gemini Pro Vision} \cite{gemini2024}: Model de Google amb bones capacitats visuals, per\`o amb limitacions de quota en el tier gratu\"it.
\end{itemize}

El projecte \textbf{DiagramLens} \cite{diagramlens} va ser pioner en l'aplicaci\'o de visi\'o multimodal a la descripci\'o de diagrames UML, generant descripcions t\`ecniques que poden ser utilitzades com a evid\`encia en l'avaluaci\'o. Evaluaitor-Lamb integra i amplia aquesta l\`ogica dins del seu sistema de tools.

\chapter{Justificaci\'o}
\label{chap:justificacio}

Per donar soluci\'o als problemes de rigidesa i manca d'adaptabilitat identificats en la secci\'o anterior, s'han avaluat diferents alternatives t\`ecniques abans de decidir el rumb d'aquest projecte. La decisi\'o fonamental recau en com abordar la transformaci\'o del pipeline existent de SE-rubric-evaluAItor.

A continuaci\'o, s'exposa l'an\`alisi comparatiu de les alternatives considerades:

\textbf{Alternativa A: Mantenir i millorar el pipeline seq\"uencial existent}

La primera opci\'o consistiria a mantenir l'arquitectura actual de SE-rubric-evaluAItor com un script Python seq\"uencial, afegint-hi millores com configuraci\'o externa de r\'ubriques, processament per lots i integraci\'o del m\`odul de visi\'o de DiagramLens.

\begin{itemize}
    \item \textbf{Avantatges:} Baix cost de desenvolupament, aprofita tot el codi existent, f\`acil de provar i validar.
    \item \textbf{Inconvenients:} No resol el problema de fons de la rigidesa. El flux d'execuci\'o segueix sent predefinit i no s'adapta al contingut de cada document. A m\'es, la manca d'un sistema d'eines din\`amics limita la capacitat de reutilitzaci\'o per a altres tipus d'entregables.
\end{itemize}

\textbf{Alternativa B: Desenvolupar un microservei d'avaluaci\'o (Evaluaitor original)}

La segona opci\'o era la planificaci\'o original del TFG: crear un microservei amb API REST, cues as\'incrones, sistema de plugins i multi-tenan\c{c}a, integrat amb l'ecosistema LAMB via Docker Compose.

\begin{itemize}
    \item \textbf{Avantatges:} Arquitectura robusta i escalable, separaci\'o neta de responsabilitats, processament as\'incron fiable.
    \item \textbf{Inconvenients:} Alt cost de desenvolupament, acoblament a l'arquitectura espec\'ifica de LAMB (que est\`a en constant canvi), requereix infraestructura de desplegament i coordinaci\'o amb l'equip LAMB. La complexitat excedeix l'abast raonable d'un TFG individual en un entorn tan din\`amic.
\end{itemize}

\textbf{Alternativa C: Transformar en una Skill ag\`entica per a opencode.ai}

La tercera alternativa era convertir el pipeline r\'igid en una \textbf{Skill ag\`entica} per a la plataforma opencode.ai, que permeti a un agent d'IA seleccionar din\`amicament les eines d'extracci\'o, an\`alisi i avaluaci\'o segons el contingut del document, amb execuci\'o local de models via Ollama.

\begin{itemize}
    \item \textbf{Avantatges:} M\`axima flexibilitat i adaptabilitat: l'agent decideix l'ordre d'execuci\'o segons el contingut. Desacoblament total de l'arquitectura de LAMB: la Skill opera de manera independent. Privacitat RGPD garantida mitjan\c{c}ant prove\"idors locals Open Source. Integraci\'o de visi\'o multimodal per a diagrames UML. C\`alcul determin\'istic de notes que elimina al·lucinacions num\`eriques. Configuraci\'o externa de r\'ubriques per a reutilitzaci\'o.
    \item \textbf{Inconvenients:} Requereix aprendre i integrar amb una plataforma nova (opencode.ai). La qualitat de l'avaluaci\'o dep\`en de la capacitat del LLM local per comprendre i aplicar les regles de la Skill. Depend\`encia de plataforma limita la portabilitat.
\end{itemize}

\textbf{Alternativa D: Sistema meta-ag\`entic complet amb agent loop propi --- \textit{Soluci\'o Final Adoptada}}

La quarta alternativa, i finalment adoptada, consisteix en implementar un \textbf{sistema meta-ag\`entic complet} amb les seg\"uents caracter\'istiques:

\begin{itemize}
    \item \textbf{Agent loop propi:} Implementaci\'o d'un agent loop independent que no dep\`en de cap plataforma externa, seguint la metodologia ``Agents All the Way Down'' \cite{alier2025} del professor Marc Alier i altres.
    \item \textbf{Workflow generator/executor:} Dos components independents que permeten generar workflows JSON personalitzats per a cada r\'ubrica i executar-los amb m\'ultiples documents, optimitzant costos i temps.
    \item \textbf{18 tools especialitzades:} Registry complet amb tools d'extracci\'o (docx\_extract, xlsx\_to\_markdown, describe\_diagrams, read\_file, list\_directory), an\`alisi (extract\_objectives, detect\_orphans, evaluate\_smart), avaluaci\'o (criterion\_evaluator, grader, report\_generator) i workflow (generate\_workflow, execute\_workflow).
    \item \textbf{DashScope i suport multi-provider:} DashScope (Alibaba Cloud) com a prove\"idor primari, amb suport addicional per a OpenRouter i Google Gemini mitjan\c{c}ant variables d'entorn, garantint resili\`encia davant limitacions de quota o disponibilitat d'un \'unic servei.
    \item \textbf{Avaluaci\'o batch de criteris:} Reducci\'o del nombre de crides a l'API mitjan\c{c}ant l'avaluaci\'o de m\'ultiples criteris en un \'unic prompt, configurada via \texttt{EVAL\_BATCH\_SIZE} (5 criteris per defecte).
    \item \textbf{Seguretat al scaffolding:} Implementaci\'o de security policy amb allow-list al scaffolding (no al prompt), seguint les bones pr\`actiques de la metodologia.
\end{itemize}

\begin{itemize}
    \item \textbf{Avantatges:} M\`axima flexibilitat i portabilitat: el sistema no dep\`en de cap plataforma externa i pot executar-se en qualsevol entorn Python. Millor qualitat d'avaluaci\'o gr\`acies als models cloud. Suport multi-provider que garanteix resili\`encia. Avaluaci\'o batch que optimitza costos d'API. Reutilitzaci\'o de workflows que optimitza costos i temps. Seguretat robusta al scaffolding. Validaci\'o rigorosa amb 6 escenaris de test (tots completats amb \`exit).
    \item \textbf{Inconvenients:} Requereix connexi\'o a internet per als prove\"idors cloud (no \'es 100\% local com Ollama). Major complexitat d'implementaci\'o respecte a una Skill simple.
\end{itemize}

\textbf{Conclusi\'o de la decisi\'o}

Avaluades totes les alternatives, es justifica l'adopci\'o de l'Alternativa D (sistema meta-ag\`entic complet). Aquesta soluci\'o aborda directament totes les limitacions identificades: rigidesa del pipeline, manca de workflows reutilitzables, depend\`encia de plataforma, i limitacions de models locals. La implementaci\'o d'un agent loop propi proporciona m\`axima flexibilitat i control, mentre que la separaci\'o entre workflow generator i executor permet optimitzar costos i temps. La migraci\'o a prove\"idors cloud millora significativament la qualitat d'extracci\'o i avaluaci\'o, el suport multi-provider garanteix resili\`encia, i l'avaluaci\'o batch redueix el cost d'API. La implementaci\'o de seguretat al scaffolding garanteix robustesa. A m\'es, la validaci\'o amb 6 escenaris reals (tots completats amb \`exit) demostra la viabilitat i robustesa del sistema en casos d'\'us reals.

\textbf{Depend\`encia de serveis cloud}

\begin{itemize}
    \item Impacte: El sistema dep\`en de la disponibilitat i quota dels prove\"idors d'API de models d'IA. Si un servei no est\`a disponible o s'exhaureix la quota, el sistema no pot funcionar amb aquest prove\"idor.

El suport multi-provider (DashScope, OpenRouter, Gemini) permet canviar de prove\"idor quan un presenta limitacions, sense modificar la l\`ogica d'avaluaci\'o. Monitoritzar el consum de tokens i establir l\'imits per evitar l'esgotament de la quota.
\end{itemize}

\textbf{Calendari apretat}

\begin{itemize}
    \item Impacte: A causa de la naturalesa del projecte, ja que \'es un treball de final de carrera, el temps de desenvolupament \'es relativament curt i aix\`o pot influir directament en la qualitat del producte final.
    \item Mitigaci\'o: Establir una planificaci\'o el m\'es acurada possible des del principi de les tasques a realitzar en un abast raonable segons les hores planificades per al projecte.
\end{itemize}

\textbf{P\`erdua d'estructura en l'extracci\'o de PDFs}

\begin{itemize}
    \item Impacte: La conversi\'o de PDF a Markdown pot perdre jerarquia d'encap\c{c}alaments, taules o llistes, afectant la qualitat de l'extracci\'o posterior.
    \item Mitigaci\'o: Utilitzar PyMuPDF amb fallback a pytesseract per a OCR, i avaluar la integraci\'o de Docling com a motor principal en fases futures.
\end{itemize}

    \item \textbf{Fase 4 (Refinament i Documentaci\'o):} Implementaci\'o de millores identificades (configuraci\'o YAML, gesti\'o de fitxers intermedios), proves amb m\'ultiples documents, i redacci\'o de la mem\`oria final.
\end{enumerate}

Aquest enfocament ha pem\'es tenir versions funcionals del sistema des de les primeres fases, facilitant l'adaptaci\'o a possibles canvis en els requisits del projecte. El control de versions del codi s'ha gestionat mitjan\c{c}ant Git, integrant els canvis progressivament per evitar trencaments en la branca principal, seguint el flux de treball de \textit{GitFlow}.

\subsection{Capa d'abstracci\'o de models (Clients LLM)}
\label{subsec:clients-llm}
Per garantir la m\`axima flexibilitat i complir de forma estricta amb el requisit de suport multi-prove\"idor (fonamental per assegurar la privacitat de les dades acad\`emiques via models locals), el sistema implementa una arquitectura basada en interf\'icies per a la connexi\'o amb els models d'IA:
\begin{itemize}
    \item \textbf{\texttt{BaseLLMClient} (\texttt{core/clients/base.py}):} Defineix la interf\'icie abstracta que tots els clients han d'implementar. Estandarditza m\`etodes clau com \texttt{chat()} per al suport de \textit{function calling} (tool calls), \texttt{generate()} per a text pla, i \texttt{vision()} per a l'an\`alisi d'imatges i diagrames. Tamb\'e centralitza les estructures de dades comunes \texttt{ToolCall} i \texttt{ChatResponse}.
    \item \textbf{\texttt{DashScopeClient}:} Implementaci\'o concreta per connectar amb l'API d'Alibaba Cloud, donant acc\'es als models de la fam\'ilia Qwen.
    \item \textbf{\texttt{OllamaClient}:} Implementaci\'o nativa per connectar amb daemons locals d'Ollama. Suporta tant models de llenguatge (ex. \texttt{qwen2.5-coder}, \texttt{llama3}) com models de visi\'o multimodal (ex. \texttt{llava}, \texttt{minicpm-v}), traduint de forma transparent les respostes locals a les estructures de dades esperades per l'agent.
\end{itemize}
El component \texttt{Agent} utilitza el patr\'o de disseny Factory durant la seva instanciació, llegint la variable d'entorn \texttt{LLM\_PROVIDER} per injectar el client corresponent. Aquest disseny permet commutar entre l'execuci\'o al n\'uvol o en local modificant nom\'es un fitxer \texttt{.env}, sense alterar absolutament res de la l\`ogica interna d'avaluaci\'o.

Les fases del flux s\'on les seg\"uents:

\begin{enumerate}
    \item \textbf{Generaci\'o de workflow (Fase 1):} L'usuari proporciona una r\'ubrica i el \texttt{Workflow Generator} analitza els criteris i genera un workflow JSON personalitzat que especifica quines tools executar i en quin ordre. Aquest workflow \'es reutilitzable amb m\'ultiples documents. La r\'ubrica es pot proporcionar en tres formats: \textbf{YAML} (format natiu del sistema), \textbf{Markdown} (la tool \texttt{rubric\_importer} la converteix autom\`aticament a YAML), o \textbf{Excel} (la tool \texttt{xlsx\_to\_markdown} la converteix primer a Markdown i despr\'es a YAML). Aquesta flexibilitat permet als docents utilitzar les r\'ubriques en el format que ja tenen, sense necessitat de conversi\'o manual.
    \item \textbf{Execuci\'o de workflow (Fase 2):} L'usuari proporciona un document i el workflow. El \texttt{Workflow Executor} resol les variables d'entorn (\texttt{\$\{env.VAR\}}) i executa els steps seq\"uencialment:
    \begin{itemize}
        \item \textbf{Extracci\'o:} Les tools \texttt{docx\_extract} i \texttt{xlsx\_to\_markdown} converteixen el document a Markdown, cridant el LLM per a tasques d'OCR si cal.
        \item \textbf{Visi\'o multimodal:} La tool \texttt{describe\_diagrams} envia les imatges al model de visi\'o i obt\'e descripcions t\`ecniques dels diagrames UML.
        \item \textbf{Avaluaci\'o batch:} La tool \texttt{criterion\_evaluator} agrupa els criteris en batches de 5, enviant m\'ultiples criteris en un \'unic prompt al LLM. Si el batch parsing falla, es rebaixa autom\`aticament a avaluaci\'o individual (Secci\'o~\ref{subsec:batch-eval}).
    \end{itemize}
    \item \textbf{C\`alcul determin\'istic (Fase 3):} El \texttt{Grader} rep les puntuacions parcials i els pesos de la r\'ubrica (internament en format YAML), i calcula la nota final ponderada mitjan\c{c}ant un script Python, sense intervenci\'o del LLM.
    \item \textbf{Generaci\'o d'informe (Fase 4):} El sistema genera un informe Markdown estructurat amb la taula de r\'ubrica, an\`alisi per criteri amb evid\`encies vinculades, i recomanacions accionables.
\end{enumerate}

Aquest flux permet separar la generaci\'o de workflows de la seva execuci\'o, optimitzant costos i temps de processament. La separaci\'o entre Fase 1 i Fase 2 \'es clau: un workflow generat una vegada pot reutilitzar-se per avaluar m\'ultiples documents del mateix tipus, estalviant la crida al LLM per a la generaci\'o del workflow en cada avaluaci\'o.

\subsection{Resoluci\'o de variables en workflows}

El workflow executor (\texttt{WorkflowExecutor}) resol refer\`encies a variables d'entorn dins dels workflows JSON mitjan\c{c}ant el patr\'o \texttt{\$\{env.VAR\_NAME\}}. Per exemple, un workflow pot especificar:

\begin{lstlisting}[language=JSON]
{
  "model": "${env.DASHSCOPE_MODEL}",
  "vision_model": "${env.DASHSCOPE_VISION_MODEL}"
}
\end{lstlisting}

La funci\'o \texttt{\_resolve\_variables} implementa la resoluci\'o en dues fases: primer les variables d'entorn (\texttt{\$\{env.\dots\}}) i despr\'es les refer\`encies a resultats de steps anteriors (\texttt{\$\{step\_id.result.key\}}):

\begin{lstlisting}[language=Python]
ENV_PATTERN = re.compile(r"\$\{env\.([^}]+)\}")

def _resolve_variables(self, value):
    if isinstance(value, str):
        # Phase 1: resolve env vars
        value = ENV_PATTERN.sub(
            lambda m: os.environ.get(m.group(1), m.group(0)),
            value
        )
        # Phase 2: resolve step references
        if VAR_PATTERN.fullmatch(value):
            var_path = VAR_PATTERN.search(value).group(1)
            parts = var_path.split(".")
            if len(parts) >= 3 and parts[1] in ("output", "result"):
                step_id, key = parts[0], ".".join(parts[2:])
                val = self.step_results[step_id].get("result", {})
                for k in key.split("."):
                    if isinstance(val, dict):
                        val = val.get(k)
                    else:
                        return None
                return val  # preserve type (dict, list, etc.)
            elif var_path in self.variables:
                return self.variables[var_path]
        # Partial interpolation
        return VAR_PATTERN.sub(replacer, value)
    elif isinstance(value, dict):
        return {k: self._resolve_variables(v)
                for k, v in value.items()}
    elif isinstance(value, list):
        return [self._resolve_variables(v) for v in value]
    return value
\end{lstlisting}

Aquesta implementaci\'o preserva el tipus de dades quan el valor \'es una refer\`encia completa (retorna diccionaris i llistes tal qual), i converteix a cadena nom\'es en el cas d'interpolaci\'o parcial, permetent que el mateix workflow sigui reutilitzable amb diferents prove\"idors i models sense modificaci\'o.

\section{Avaluaci\'o batch de criteris}
\label{subsec:batch-eval}

\subsection{Motivaci\'o}

L'avaluaci\'o individual de criteris (una crida a l'API per criteri) \'es l'enfocament m\'es senzill per\`o presentava un problema pr\`actic greu: amb 13 criteris per mem\`oria i 6 mem\`ories a avaluar, es requerien 78 crides a l'API, excedint r\`apidament les quotes dels prove\"idors gratu\"its. Per exemple, OpenRouter limita el tier gratu\"it a 50 crides/dia, i Google Gemini a 20 crides/minut.

\subsection{Disseny de la soluci\'o}

Es va implementar un mecanisme d'avaluaci\'o batch que agrupa m\'ultiples criteris en un \'unic prompt, demanant al LLM que avalu\"i tots els criteris del batch simult\`aniament i retorni les puntuacions en un format JSON estructurat. El sistema mant\'e dues vies d'execuci\'o:

\begin{enumerate}
    \item \textbf{Via batch (\texttt{evaluate\_criteria\_batch}):} Agrupa criteris no-visuals en batches de mida configurable (\texttt{EVAL\_BATCH\_SIZE}, per defecte 5). Construeix un prompt que inclou tots els criteris del batch i les seves descripcions, i demana al LLM que retorni un JSON amb les puntuacions i evid\`encies per a cadascun.
    \item \textbf{Via individual (\texttt{\_evaluate\_single}):} Avalua un \'unic criteri per crida. S'utilitza per a criteris que requereixen visi\'o multimodal (on el context visual no es pot compartir entre criteris) o quan el batch parsing falla.
\end{enumerate}

\subsection{Construcci\'o del prompt batch}

La funci\'o \texttt{\_build\_batch\_prompt} construeix un \'unic prompt que inclou tots els criteris del batch, els seus nivells d'avaluaci\'o (extrets del fitxer de prompt de cada criteri), el document complet i el context d'an\`alisi previ, i demana al LLM una resposta estructurada amb delimitadors:

\begin{lstlisting}[language=Python]
def _build_batch_prompt(criteria_batch, document, context_str=None):
    context_section = context_str or (
        "No hay analisis previo disponible."
    )
    criteria_descriptions = []
    for i, c in enumerate(criteria_batch, 1):
        prompt_template = load_prompt(c.prompt)
        levels_text = ""
        for line in prompt_template.split("\n"):
            if line.strip().startswith("- **"):
                levels_text += line + "\n"
        criteria_descriptions.append(
            f"### Criterio {i}: {c.name} (ID: {c.id})\n"
            f"Niveles:\n{levels_text}"
        )
    criteria_block = "\n".join(criteria_descriptions)
    prompt = f"""Analiza el documento y evalua CADA
UNO de los criterios listados.

Documento:
{document}

## Contexto de Analisis Previo
{context_section}

## Criterios a evaluar
{criteria_block}

## Formato de respuesta
Para CADA criterio, usa EXACTAMENTE:
---CRITERIO_START---
# Evaluacion: [NOMBRE DEL CRITERIO]
## Puntuacion
**Puntuacion:** X/10
---CRITERIO_END---"""
    return prompt
\end{lstlisting}

L'\'us de delimitadors (\texttt{---CRITERIO\_START---} / \texttt{---CRITERIO\_END---}) permet al parser identificar inequ\'ivocament el bloc de cada criteri, independentment de la longitud del text generat per al an\`alisi o les observacions.

\subsection{Reducci\'o de crides a l'API}

Amb una r\'ubrica de 13 criteris i \texttt{EVAL\_BATCH\_SIZE=5}:

\begin{center}
$\lceil 13 / 5 \rceil = 3$ crides API (en lloc de 13) $\Rightarrow$ \textbf{reducci\'o del 77\%}
\end{center}

Per a l'avaluaci\'o de 6 mem\`ories:

\begin{center}
$6 \times 3 = 18$ crides (en lloc de 78) $\Rightarrow$ \textbf{reducci\'o del 77\%}
\end{center}

\subsection{Parsing de la resposta batch}

El parser de respostes batch (\texttt{\_parse\_batch\_response}) extreu les puntuacions del text retornat pel LLM utilitzant els delimitadors com a marca de separaci\'o entre criteris:

\begin{lstlisting}[language=Python]
def _parse_batch_response(text, criteria_ids):
    results = {}
    # Split by CRITERIO_START delimiter
    blocks = re.split(r"---CRITERIO_START---", text)
    for block in blocks[1:]:
        block = block.strip()
        # Truncate at CRITERIO_END
        end_idx = block.find("---CRITERIO_END---")
        if end_idx != -1:
            block = block[:end_idx].strip()
        # Match block to criterion ID
        criterion_id = None
        for cid in criteria_ids:
            normalized = cid.replace("_", " ")
            if normalized in block.lower() or cid in block:
                criterion_id = cid
                break
        # Fallback: positional matching
        if criterion_id is None and criteria_ids:
            idx = len([k for k in results
                       if k in criteria_ids])
            if idx < len(criteria_ids):
                criterion_id = criteria_ids[idx]
        if criterion_id:
            results[criterion_id] = block
    # Graceful degradation: assign 0.0 to missing
    for cid in criteria_ids:
        if cid not in results:
            results[cid] = (
                f"# Evaluacion: {cid}\n"
                f"## Puntuacion\n**Puntuacion:** 0/10\n"
                f"## Observaciones\nNo se pudo extraer."
            )
    return results
\end{lstlisting}

El parser implementa tres estrat\`egies de coincid\`encia per identificar cada bloc amb el seu criteri:

\begin{enumerate}
    \item \textbf{Coincid\`encia per ID normalitzat:} Busca l'ID del criteri (amb subratllats substitu\"its per espais) dins del bloc de text.
    \item \textbf{Coincid\`encia posicional:} Si no es troba l'ID, assigna el bloc a la seg\"uent posici\'o no ocupada, seguint l'ordre original dels criteris.
    \item \textbf{Degradaci\'o graceful:} Si un criteri no t\'e bloc corresponent, s'assigna autom\`aticament una puntuaci\'o de 0/10 amb una observaci\'o que indica que no es va poder extreure l'avaluaci\'o del batch.
\end{enumerate}

Aquest mecanisme de \textit{graceful degradation} garanteix que el sistema sempre produeix un resultat complet, encara que el LLM no segueixi exactament el format esperat.

\section{Gesti\'o d'errors i retry logic}
\label{subsec:retry-logic}

Durant el desenvolupament i les proves del sistema amb m\'ultiples prove\"idors d'IA, es va identificar que les crides a les APIs cloud s\'on propenses a errors transitoris: l\'imits de ritme (\textit{rate limiting}, HTTP 429), errors de servidor (5xx), timeouts, i respostes mal formatades. Aquests errors no s\'on excepcionals sin\'o esperables en l'\'us quotidi\`a del sistema, especialment amb prove\"idors gratu\"its que imposeen restriccions severes (OpenRouter: 50 crides/dia; Gemini: 20 crides/minut).

Per garantir la fiabilitat del sistema, es va implementar un mecanisme de reintent autom\`atic (\textit{retry logic}) amb les seg\"uents caracter\'istiques:

\subsection{Classificaci\'o d'errors}

El sistema classifica les respostes HTTP en tres categories:

\begin{enumerate}
    \item \textbf{Errors reintentables:} HTTP 408 (Request Timeout), 429 (Too Many Requests), 500 (Internal Server Error), 502 (Bad Gateway), 503 (Service Unavailable), 504 (Gateway Timeout). Aquests errors indiquen problemes transitoris que poden resoldre's esperant i reintentant.
    \item \textbf{Errors no reintentables:} HTTP 401 (Unauthorized), 403 (Forbidden), 404 (Not Found). Aquests errors indiquen problemes permanents (clau inv\`alida, model no disponible) que no es resoldran reintentant.
    \item \textbf{Cas especial --- HTTP 400 (Bad Request):} Es classifica com a reintentable perqu\`e alguns prove\"idors retornen 400 per a errors de quota o de format que poden ser transitoris (p.ex., quan el payload supera el l\'imit de tokens del context).
\end{enumerate}

\subsection{Parsing del delay suggerit pel servidor}

Una millora clau respecte al retry logic b\`asic \'es la capacitat de \textbf{llegir el delay suggerit pel servidor} en les respostes HTTP 429. Diferents prove\"idors inclouen missatges com:

\begin{lstlisting}[language=bash]
# Google Gemini
"Please retry in 30.4s"

# OpenRouter  
"Rate limit exceeded. Please retry after 60s"
\end{lstlisting}

El client implementa un parser d'expressions regulars que extreu el delay suggerit:

\begin{lstlisting}[language=Python]
import re
match = re.search(r"retry in ([\d.]+)\s*s", error_text, re.IGNORECASE)
if match:
    suggested_delay = float(match.group(1))
    wait_time = suggested_delay + 2.0  # marge de seguretat
\end{lstlisting}

Quan el servidor proporciona un delay suggerit, el sistema l'utilitza directament (amb un marge de 2 segons addicionals per seguretat), en lloc d'aplicar el backoff exponencial. Aix\`o evita dues situacions problem\`atiques:

\begin{itemize}
    \item \textbf{Reintents massa agressius:} Si el servidor indica ``retry in 60s'' i el backoff exponencial nom\'es hauria esperat 4s, el sistema podria esgotar tots els reintents abans que el l\'imit s'aixequ\'es.
    \item \textbf{Esperes innecess\`ariament llargues:} Si el servidor indica ``retry in 2s'' i el backoff hauria esperat 60s, el sistema perd temps innecess\`ariament.
\end{itemize}

\subsection{Backoff exponencial amb jitter}

Quan el servidor no proporciona un delay suggerit (errors 5xx, timeouts), el sistema aplica backoff exponencial amb un petit jitter aleatori per evitar la sincronitzaci\'o de reintents:

\begin{lstlisting}[language=Python]
delay = min(max_delay, delay * backoff_factor)
wait_time = delay + (0.1 * (attempt + 1))
\end{lstlisting}

\label{tab:backoff}
\begin{tabular}{@{}ccc@{}}
\toprule
\textbf{Intent} & \textbf{Delay base (s)} & \textbf{Amb jitter (s)} \\
\midrule
1 & 1{,}0 & 1{,}1 \\
2 & 2{,}0 & 2{,}2 \\
3 & 4{,}0 & 4{,}3 \\
4 & 8{,}0 & 8{,}4 \\
5 & 16{,}0 & 16{,}5 \\
\bottomrule
\end{tabular}
\end{table}

\subsection{Protecci\'o contra payloads excessius}

El client implementa una comprovaci\'o de mida m\`axima del payload abans d'enviar la petici\'o, amb un l\'imit de 10 MB (especialment rellevant per a crides de visi\'o que inclouen imatges codificades en base64). Si el payload supera aquest l\'imit, es genera un error immediatament sense consumir una crida a l'API:

\begin{lstlisting}[language=Python]
MAX_VISION_PAYLOAD_SIZE = 10 * 1024 * 1024  # 10 MB
payload_size = len(json.dumps(payload).encode("utf-8"))
if payload_size > MAX_VISION_PAYLOAD_SIZE:
    raise ValueError(f"Payload exceeds limit.")
\end{lstlisting}

\subsection{Resultats pr\`actics}

Durant la validaci\'o del sistema, el retry logic ha demostrat ser essencial per a l'operabilitat amb prove\"idors gratu\"its. Es van observar els seg\"uents patrons:

\begin{itemize}
    \item \textbf{Google Gemini} retorna freq\"uentment HTTP 429 amb el missatge ``Please retry in Xs''. Sense el parser de delay suggerit, el sistema es quedava bloquejat esgotant els reintents. Amb el parser, el sistema espera el temps exacte i reeix en el seg\"uent intent.
    \item \textbf{OpenRouter} presenta l\'imits diaris (50 crides/dia) que no es poden resoldre amb reintents, per\`o el retry logic gestiona correctament els errors 429 intermitents dins del l\'imit diari.
    \item \textbf{DashScope} ocasionalment retorna HTTP 500 o 503 per sobrec\`arrega del servei, que es resolen autom\`aticament amb 1--2 reintents.
\end{itemize}

\subsection{Discussi\'o}

A partir de les avaluacions completades fins a la data, es poden identificar els seg\"uents patrons preliminars:

\begin{enumerate}
    \item \textbf{Biaix sistem\`atic cap a la infra-avaluaci\'o:} L'agent infra-avalua respecte a l'avaluadora humana en 5 de 6 mem\`ories, amb una desviaci\'o mitjana de $\bar{\Delta} = -2,06$ punts. L'\'unica excepci\'o \'es l'Alumne 6, on l'agent sobre-avalua lleugerament ($+0,63$). Aix\`o indica que el model \texttt{gpt-oss-120b:free} t\'e una tend\`encia sistem\`atica a assignar puntuacions m\'es baixes que un expert hum\`a.

    \item \textbf{Correlaci\'o baixa entre avaluadors:} La correlaci\'o de Pearson entre les notes de l'agent i l'avaluadora humana \'es $r = 0,226$, la qual cosa indica una correlaci\'o positiva per\`o feble. Aix\`o suggereix que l'agent i l'hum\`a no coincideixen en la ordenaci\'o relativa de les mem\`ories: l'avaluadora humana assigna notes altes (9,71--10,0) a 3 mem\`ories que l'agent puntua entre 5,23 i 6,23.

    \item \textbf{Major variabilitat en criteris operatius:} Els criteris relacionats amb accions concretes (millores de 1a part, recursos did\`actics, perspectiva de g\`enere) presenten major dispersi\'o que els criteris reflexius (comprensi\'o, observaci\'o, reflexi\'o).

    \item \textbf{MAE de 2,27 punts:} La desviaci\'o absoluta mitjana indica que, de mitjana, l'agent es desvia 2,27 punts de la nota de l'avaluadora humana. La desviaci\'o m\`axima \'es de 4,48 punts (Alumne 4) i la m\'inima de 0,63 (Alumne 6).

    \item \textbf{Limitaci\'o de la mostra:} Amb 6 mem\`ories completes, les m\`etriques estad\'istiques tenen major pot\`encia que amb les 2 inicials, per\`o es requeriria un m\'inim de 8--10 mem\`ories per obtenir resultats estad\'isticament significatius. A m\'es, la comparaci\'o es realitza nom\'es a nivell de nota final, no per criteri, degut a la disponibilitat limitada de les puntuacions de l'avaluadora.
\end{enumerate}

Les m\`etriques de comparaci\'o amb l'avaluadora humana ($\bar{\Delta} = -2,06$, MAE $= 2,27$, correlaci\'o de Pearson $r = 0,226$) indiquen que l'agent tendeix a infra-avaluar respecte a l'expert hum\`a i que la correlaci\'o entre ambd\'os avaluadors \'es feble, la qual cosa significa que l'agent i l'experta no coincideixen en l'ordre relatiu (o r\`anquing) de les mem\`ories avaluades.

\section{Conclusions de la validaci\'o}
\label{sec:conclusions-validacio}

La validaci\'o del sistema Evaluaitor-Lamb ha demostrat que:

\begin{enumerate}
    \item \textbf{El workflow generator funciona correctament:} Genera workflows JSON v\`alids i reutilitzables per a cada r\'ubrica.
    \item \textbf{El workflow executor funciona correctament:} Executa workflows existents amb documents espec\'ifics, gestionant variables i errors.
    \item \textbf{El session management funciona correctament:} Permet converses multi-turn amb continuaci\'o de sessi\'o i canvis manuals.
    \item \textbf{La seguretat \'es robusta:} Rebutja inputs maliciosos de manera segura i professional, oferint alternatives educatives.
    \item \textbf{La visi\'o multimodal funciona correctament:} Ha processat 25 de 26 imatges amb \`exit (96\%), millorant significativament la qualitat de l'avaluaci\'o de diagrames UML.
    \item \textbf{El flux complet funciona:} L'Escenari 1 ha validat el flux complet del sistema (generaci\'o + execuci\'o + informe) amb una nota de 7.0/10.
    \item \textbf{Validaci\'o comparativa amb avaluador hum\`a:} La comparaci\'o amb una avaluadora humana experta sobre 6 mem\`ories reals de m\`aster ha revelat un biaix sistem\`atic cap a la infra-avaluaci\'o i una correlaci\'o encara feble (Secci\'o~\ref{sec:validacio-humana}). L'agent genera un rang de notes entre 4,77 i 7,54, mentre que l'avaluadora humana assigna entre 6,00 i 10,00. L'agent tendeix a infra-avaluar ($\bar{\Delta} = -2,06$), amb un MAE de 2,27 punts i una correlaci\'o de Pearson de $r = 0,226$.
\end{enumerate}

En resum, la validaci\'o t\`ecnica del sistema Evaluaitor-Lamb \'es robusta en els 6 escenaris provats; no obstant aix\`o, la comparaci\'o amb l'experta revela un biaix sistem\`atic cap a la infra-avaluaci\'o i una correlaci\'o encara feble, la qual cosa marca el cam\'i per a la calibraci\'o futura de l'eina abans del seu desplegament en producci\'o.

\section{Resum de les contribucions}
\label{sec:contribucions}

Aquest Treball de Fi de Grau ha presentat el disseny, implementaci\'o i validaci\'o de Evaluaitor-Lamb, un sistema meta-ag\`entic per a l'avaluaci\'o assistida d'entregables acad\`emics. El sistema est\`a dissenyat per a l'avaluaci\'o multi-document. Tot i que els escenaris de validaci\'o de la present mem\`oria se centren en les mem\`ories finals de pr\`actiques de m\`aster (per ser un lliurable ric que combina text, dades d'Excel i diagrames), les eines i els workflows s'han dissenyat de manera modular i desacoblada, permetent avaluar qualsevol altre document acad\`emic adaptant exclusivament el fitxer de r\'ubrica YAML.

Les contribucions principals del projecte s\'on les seg\"uents:

\begin{enumerate}
    \item \textbf{Agent loop propi independent:} Implementaci\'o d'un agent loop que gestiona el cicle de vida de l'agent sense dependre de cap plataforma externa, seguint la metodologia ``Agents All the Way Down'' \cite{alier2025} del professor Marc Alier i altres. L'agent loop compleix la disciplina KV-cache (tools i system est\`atics, messages en append-only) i implementa el Turtle Pattern com a CLI natiu de terminal.

    \item \textbf{Workflow generator/executor:} Dos components desacobllats que permeten generar workflows JSON personalitzats per a cada r\'ubrica i reutilitzar-los amb m\'ultiples documents. Aquesta separaci\'o optimitza costos d'API i temps de processament: un workflow generat una vegada pot executar-se amb tants documents com calgui sense regenerar-lo.

    \item \textbf{Suport multi-provider:} Una capa d'abstracci\'o que permet intercanviar prove\"idors de models d'IA (DashScope, OpenRouter, Google Gemini, Ollama local) mitjan\c{c}ant variables d'entorn (\texttt{DASHSCOPE\_BASE\_URL}, \texttt{DASHSCOPE\_MODEL},\\ \texttt{DASHSCOPE\_VISION\_MODEL}), sense modificar la l\`ogica d'avaluaci\'o. Aquest disseny garanteix resili\`encia davant limitacions de quota i disponibilitat dels serveis cloud.

    \item \textbf{Avaluaci\'o batch de criteris:} Un mecanisme que agrupa m\'ultiples criteris en un \'unic prompt al LLM, reduint les crides a l'API en un 77\% (de 13 a 3 crides per mem\`oria amb \texttt{EVAL\_BATCH\_SIZE=5}). El parser de respostes implementa graceful degradation amb tres estrat\`egies de coincid\`encia, garantint que el sistema sempre produeix un resultat complet.

    \item \textbf{C\`alcul determin\'istic de notes amb tra\c{c}abilitat d'evid\`encia:} La nota final es calcula exclusivament mitjan\c{c}ant scripts Python, eliminant les al·lucinacions num\`eriques dels LLMs. Cada judici evaluatiu est\`a vinculat a evid\`encia textual o visual del document, permetent l'auditoria i la transpar\`encia del proc\'es d'avaluaci\'o.

    \item \textbf{18 tools especialitzades:} Un registry centralitzat amb tools d'extracci\'o (docx\_extract, xlsx\_to\_markdown, describe\_diagrams, read\_file, list\_directory), an\`alisi (extract\_objectives, detect\_orphans, evaluate\_smart, classify\_iso25010), avaluaci\'o (criterion\_evaluator, grader, report\_generator), configuraci\'o (rubric\_importer) i workflow (generate\_workflow, execute\_workflow).

    \item \textbf{Visi\'o multimodal integrada:} Integraci\'o de la l\`ogica de DiagramLens per a la descripci\'o i categoritzaci\'o de diagrames UML utilitzant models de visi\'o, proporcionant evid\`encia visual per a l'avaluaci\'o de consist\`encia arquitect\`onica. Validada amb 96\% d'\'exit (25/26 imatges processades).

    \item \textbf{Gesti\'o d'errors robusta:} Retry logic amb backoff exponencial, parsing del delay suggerit pel servidor en respostes HTTP 429 (suportant els formats de Gemini, OpenRouter i DashScope), i protecci\'o contra payloads excessius.
\end{enumerate}

\section{Limitacions identificades}
\label{sec:limitacions-conclusions}

Tot i l'\'exit global del projecte, s'identifiquen les seg\"uents limitacions:

\begin{enumerate}
    \item \textbf{Depend\`encia de serveis cloud:} El sistema requereix connexi\'o a internet per accedir als prove\"idors de models d'IA. Encara que el suport multi-provider mitiga aquesta depend\`encia permetent failover entre prove\"idors, no \'es possible operar completament fora de l\'inia. L'\'us d'Ollama com a prove\"idor local \'es possible per\`o amb qualitat d'avaluaci\'o significativament inferior.

    \item \textbf{Quotes de l'API:} Els prove\"idors gratu\"its imposeen restriccions (OpenRouter: l\'imit diari variable segons el model; Gemini: 20 crides/minut) que limiten la capacitat d'avaluar grans volums de documents. L'avaluaci\'o batch redueix el consum en un 77\%, per\`o no l'elimina completament.

    \item \textbf{Mostra redu\"ida de validaci\'o comparativa:} La validaci\'o amb avaluadora humana s'ha completat amb 6 mem\`ories completes, per\`o la comparaci\'o nom\'es s'ha pogut realitzar a nivell de nota final (no per criteri), ja que l'avaluadora no va proporcionar les puntuacions desglossades. Es requeriria un m\'inim de 8--10 mem\`ories amb puntuacions per criteri per obtenir resultats estad\'isticament significatius.

    \item \textbf{Filtre de contingut de l'API:} El 4\% de les imatges (1 de 26) va ser rebutjada pel filtre de contingut de DashScope, perdent-se informaci\'o visual en casos molt espec\'ifics.

    \item \textbf{Qualitat variable segons el model:} Diferents models produeixen resultats d'avaluaci\'o amb diferent nivell de detall i coher\`encia. L'elecci\'o del model afecta directament la qualitat de l'avaluaci\'o, i no existeix un mecanisme autom\`atic per seleccionar el model \`optim per a cada tasca.

    \item \textbf{Fase P5 no implementada:} El behavioral testing automatitzat (Agent-Tests-Agent) queda pendent, la qual cosa significa que les validacions es realitzen manualment i no es poden executar de manera cont\'inua en un pipeline CI/CD.

    \item \textbf{Biaix d'infra-avaluaci\'o del model:} El model \texttt{gpt-oss-120b:free} tendeix a infra-avaluar sistem\`aticament respecte a un expert hum\`a ($\bar{\Delta} = -2,06$), especialment en els criteris de Filosofia docent, Comunicaci\'o efica\c{c} i Compliment/dietari (mitjanes 3,0--4,8). Aix\`o pot requerir un factor de calibratge o l'\'us d'un model alternatiu per millorar la concordan\c{c}a amb l'avaluaci\'o humana.
\end{enumerate}

\section{Treball futur}
\label{sec:treball-futur}

A partir de les contribucions i limitacions identificades, es proposen les seg\"uents l\'inies de treball futur:

\subsection{L\'inia 1: Validaci\'o comparativa ampliada}

\begin{itemize}
    \item Obtenir les puntuacions per criteri de l'avaluadora humana per completar les m\`etriques de comparaci\'o (MAE, correlaci\'o de Pearson, desviaci\'o m\`axima).
    \item Realitzar un estudi d'ablaci\'o sistem\`atic: comparar resultats amb vs sense visi\'o, amb avaluaci\'o individual vs batch, i amb diferents models (p.ex., Qwen vs Gemma vs Llama).
    \item Ampliar el conjunt de documents a altres tipus d'entregables (documents de requisits, disseny arquitect\`onic, manuals t\`ecnics) per validar la generalitzabilitat del sistema.
\end{itemize}

\subsection{L\'inia 2: Fase P5 --- Agent-Tests-Agent}

\begin{itemize}
    \item Implementar behavioral testing automatitzat seguint la fase P5 de la metodologia AAWD.
    \item Dissenyar un agent avaluador que condueixi l'agent principal amb escenaris de prova predefinits, verificant autom\`aticament que les respostes compleixen les expectatives (puntuacions en rang, evid\`encia present, format correcte).
    \item Integrar les proves en un pipeline CI/CD per a validaci\'o cont\'inua.
\end{itemize}

\subsection{L\'inia 3: Integraci\'o amb l'ecosistema LAMB}

\begin{itemize}
    \item Desacoblar l'avaluaci\'o de la interf\'icie CLI i exposar-la com a microservei amb API REST, permetent la seva integraci\'o directa amb la plataforma LAMB.
    \item Implementar cues as\'incrones per al processament per lots de m\'ultiples documents.
    \item Afegir multi-tenan\c{c}a per permetre que diferents institucions utilitzin el sistema amb les seves pr\`opies r\'ubriques i configuracions.
\end{itemize}


\section{Reflexi\'o personal}
\label{sec:reflexio}

Aquest Treball de Fi de Grau ha representat un repte significatiu a nivell t\`ecnic i metodol\`ogic. La reorientaci\'o del projecte des de la proposta inicial (microservei per a l'ecosistema LAMB) fins al sistema meta-ag\`entic final ha requerit aprendre i aplicar un conjunt de tecnologies emergents que no formaven part del curr\'iculum del grau: agents d'IA, sistemes meta-ag\`entics, visi\'o multimodal, i la metodologia ``Agents All the Way Down''.

L'experi\`encia d'implementar un agent loop propi, sense dependre de frameworks externs com LangChain o AutoGen, ha proporcionat una comprensi\'o profunda del funcionament intern dels sistemes ag\`entics. Aquesta comprensi\'o no s'hauria assolit utilitzant un framework d'alt nivell que abstragu\'es la complexitat del loop.

El desenvolupament amb prove\"idors gratu\"its d'IA ha estat un repte pr\`actic constant: les limitacions de quota, els canvis en la disponibilitat dels models, i els errors transitoris han requerit implementar mecanismes de resili\`encia (retry logic, multi-provider, avaluaci\'o batch) que no eren necessaris en un entorn de producci\'o amb acc\'es pagat. Aquestes limitacions, per\`o, han resultat en un sistema m\'es robust i resili\`ent, capa\c{c} d'operar en condicions adverses.

Finalment, la validaci\'o comparativa amb una avaluadora humana ha perm\`es comprendre no nom\'es si el sistema funciona t\`ecnicament, sin\`o si les seves avaluacions tenen sentit des de la perspectiva d'un expert hum\`a. Encara que la mostra \'es redu\"ida, els resultats preliminars s\'on encoratjadors i obren la porta a una validaci\'o ampliada com a treball futur.

\textbf{Pla B:} Si al final del Sprint 1 T1.2 no s'ha completat, es reduir\`a la dedicaci\'o a la documentaci\'o cont\'inua (T0.2) de 4h/setmana a 2h/setmana durant les setmanes 4-5, alliberant 4 hores addicionals. Afectaci\'o sobre la durada total: nul·la, ja que la documentaci\'o es recupera al Sprint 5 (T5.1 t\'e 60h reservades).

\vspace{0.5cm}

\textbf{Risc 2 -- Depend\`encia de serveis cloud}

\textbf{Impacte:} El sistema dep\`en de la disponibilitat i quota dels prove\"idors d'API de models d'IA. Si un servei no est\`a disponible o s'exhaureix la quota, el sistema no pot funcionar amb aquest prove\"idor.

\textbf{Mitigaci\'o est\`andard:} Implementar retry logic amb exponential backoff per gestionar errors temporals. Suport multi-provider (DashScope, OpenRouter, Gemini) que permet canviar de prove\"idor quan un presenta limitacions.

\textbf{Pla B:} Si la quota d'un prove\"idor s'exhaureix, es pot canviar a un prove\"idor alternatiu mitjan\c{c}ant variables d'entorn (\texttt{DASHSCOPE\_BASE\_URL}, \texttt{DASHSCOPE\_API\_KEY}) sense modificar codi. Addicionalment, l'avaluaci\'o batch redueix el nombre de crides en un 77\%, allargant la vida de la quota. Afectaci\'o: possible variaci\'o en la qualitat de l'avaluaci\'o segons el model alternatiu utilitzat, per\`o el sistema segueix sent funcional.

\vspace{0.5cm}

\textbf{Risc 3 -- Calendari apretat (15 setmanes)}

\textbf{Impacte:} Qualsevol imprevist t\`ecnic important pot comprometre l'entrega de l'MVP.

\textbf{Mitigaci\'o est\`andard:} Planificaci\'o detallada i metodologia \`agil per re-prioritzar.

\textbf{Pla B:} Si al final del Sprint 3 hi ha un retard superior a una setmana, s'activar\`a una reducci\'o d'abast controlada: s'eliminar\`a la configuraci\'o YAML externa (T4.1) i es posposar\`a la gesti\'o de fitxers intermedios (T4.2) com a treball futur.

\vspace{0.5cm}

\textbf{Risc 4 -- P\`erdua d'estructura en l'extracci\'o de PDFs}

\textbf{Impacte:} La conversi\'o de PDF a Markdown pot perdre jerarquia d'encap\c{c}alaments, taules o llistes, afectant la qualitat de l'extracci\'o posterior.

\textbf{Mitigaci\'o est\`andard:} Utilitzar PyMuPDF amb fallback a pytesseract per a OCR.

\textbf{Pla B:} Si durant les proves la qualitat d'extracci\'o \'es insuficient, s'avaluar\`a la integraci\'o de Docling com a motor principal. Aquestes hores s'obtindran reduint en 4h T5.2 i en 4h la T0.2 de les \'ultimes dues setmanes.

\section{Control de gesti\'o}
\label{sec:control-gestio}

L'objectiu del control de gesti\'o \'es comparar peri\`odicament el cost real incorregut amb el cost pressupostat, per detectar desviacions a temps i prendre mesures correctores. Es defineixen tres tipus de desviaci\'o (seguint el M\`odul 2.4):

\textbf{Desviaci\'o en tarifa} (el preu/hora real difereix de l'estimat):

\begin{center}
$\Delta$\_tarifa = (Preu estimat $-$ Preu real) $\times$ Hores reals consumides
\end{center}

\textbf{Desviaci\'o en efici\`encia} (les hores consumides difereixen de les estimades):

\begin{center}
$\Delta$\_efici\`encia = (Hores estimades $-$ Hores reals) $\times$ Preu estimat
\end{center}

\textbf{Desviaci\'o total per tasca:}

\begin{center}
$\Delta$\_total = Cost estimat tasca $-$ Cost real tasca
\end{center}

\subsection{Mecanisme de control}
\label{subsec:mecanisme-control}

El control es realitzar\`a de manera \textbf{setmanal}, coincidint amb les reunions de seguiment (T0.1), seguint aquest procediment:

\textbf{1.\ Registre de temps real (diari):} Al final de cada jornada, es registrar\`a les hores dedicades a cada tasca en un full de c\`alcul (Google Sheets).

\textbf{2.\ C\`alcul setmanal de desviacions:} Cada setmana es calcularan els indicadors seg\"uents per a cada tasca completada o en curs:

\begin{table}[H]
\centering
\caption{Indicadors de les desviacions de cada tasca. Font: Elaboraci\'o pr\`opia.}
\label{tab:indicadors}
\begin{tabularx}{\textwidth}{@{}cCc@{}}
\toprule
\textbf{Indicador} & \textbf{F\'ormula} & \textbf{Interpretaci\'o} \\
\midrule
\textbf{Desviaci\'o en tarifa ($\Delta$t)} & (Preu\_est $-$ Preu\_real) $\times$ Hores\_reals & $>$0 m\'es barat; $<$0 m\'es car \\
\textbf{Desviaci\'o en efici\`encia ($\Delta$e)} & (Hores\_est $-$ Hores\_reals) $\times$ Preu\_est & $>$0 m\'es r\`apid; $<$0 m\'es lent \\
\textbf{Desviaci\'o total ($\Delta$)} & Cost\_est $-$ Cost\_real & $>$0 estalvi; $<$0 sobrecost \\
\textbf{\% Desviaci\'o} & ($\Delta$ / Cost\_est) $\times$ 100 & Magnitud relativa \\
\bottomrule
\end{tabularx}
\end{table}

\textbf{3.\ Llindar d'alerta:} Si la desviaci\'o total acumulada supera el $\pm$10\% del pressupost en qualsevol moment del projecte, s'activar\`a una revisi\'o del pla. Per sota d'aquest llindar, la conting\`encia (2.316 \euro) absorbeix la desviaci\'o.

\textbf{4.\ Gesti\'o de desviacions:} En cas de detectar-se una desviaci\'o significativa:

\begin{itemize}
    \item \textbf{$\Delta < 10\%$:} La conting\`encia ho absorbeix. Cap acci\'o immediata.
    \item \textbf{10\% $\leq$ $\Delta < 20\%$:} Revisi\'o d'estimacions de les tasques pendents i re-planificaci\'o puntual.
    \item \textbf{$\Delta \geq$ 20\%:} Activaci\'o dels plans B corresponents i notificaci\'o al director.
\end{itemize}

\section{Dimensi\'o Ambiental}
\label{sec:dimensio-ambiental}

\textbf{Respecte el Projecte posat en producci\'o (PPP): \'Has estimat el impacte ambiental que tindr\`a la realitzaci\'o del projecte?}

El recurs principal consumit durant el desenvolupament \'es l'energia el\`ectrica del port\`atil de treball (HP 15s-eq2172ns, AMD Ryzen 5 5500U, $\sim$30W de consum real en \'us). Amb un total de 570 hores de projecte, el consum estimat \'es:

\begin{center}
0,030 kW $\times$ 570 h = 17,1 kWh
\end{center}

Utilitzant el factor d'emissi\'o del mix el\`ectric espanyol de 2024 (0,18 kg CO$_2$/kWh, font: \cite{ree}), les emissions associades a la realitzaci\'o del projecte s\'on:

\begin{center}
16,2 kWh $\times$ 0,18 kg CO$_2$/kWh = $\sim$2,92 kg CO$_2$
\end{center}

Per posar aquesta xifra en context: \'es equivalent al consum d'una nevera eficient (classe A+) encesa durant 32 hores, o a una persona en rutina habitual durant 6,7 dies. Es tracta d'un impacte ambiental molt redu\"it.

\textbf{Respecte el PPP: \'T'has plantejat minimitzar-ne el impacte, per exemple, reutilitzant recursos?}

Pel que fa a la minimitzaci\'o d'aquest impacte, el projecte utilitza prove\"idors cloud d'IA (DashScope/Alibaba Cloud com a primari, OpenRouter i Google Gemini com a alternatius), la qual cosa implica un consum energ\`etic als servidors cloud del prove\"idor. No obstant aix\`o, aquest impacte \'es compartit entre tots els usuaris de la plataforma i els models s'executen en infraestructures optimitzades per a l'efici\`encia energ\`etica. A nivell local, l'\'unic consum addicional \'es el del port\`atil de l'autor i la transmissi\'o de dades per xarxa, que \'es negligible en comparaci\'o amb l'execuci\'o local de models de gran mida. Addicionalment, els prove\"idors compleixen amb les normatives europees de protecci\'o de dades (RGPD).

\textbf{Respecte la Vida \'Util: Com es resol actualment el problema que vols abordar (estat de l'art)?, i en qu\`e millorar\`a ambientalment la teva soluci\'o respecte a les existents?}

Actualment, l'avaluaci\'o d'entregables d'Enginyeria del Software dins de l'ecosistema LAMB es realitza a trav\'es del projecte SE-rubric-evaluAItor, un pipeline seq\"uencial d'investigaci\'o. Aquest pipeline presenta limitacions estructurals: flux d'execuci\'o r\'igid i predefinit, configuraci\'o de r\'ubriques hardcodejada al codi font, sense capacitat d'an\`alisi de diagrames UML, i sense processament per lots.

Evaluaitor-Lamb resol totes aquestes limitacions transformant el pipeline en un sistema meta-ag\`entic que selecciona din\`amicament les eines segons el contingut. Ambientalment, la soluci\'o utilitza prove\"idors cloud d'IA per a l'execuci\'o dels models, la qual cosa implica un consum energ\`etic als servidors cloud del prove\"idor, per\`o aquest impacte \'es compartit entre tots els usuaris de la plataforma i s'executa en infraestructures optimitzades per a l'efici\`encia energ\`etica. Finalment, el fet de gestionar un projecte de 570 hores de manera completament aut\`onoma refor\c{c}ar\`a la meva capacitat d'organitzaci\'o i presa de decisions t\`ecniques independents.

\textbf{Respecte la Vida \'Util: Existeix una necessitat real del projecte? Com es resol actualment el problema que vols abordar? En qu\`e millorar\`a socialment la teva soluci\'o respecte les existents?}

Existeix una necessitat real i contrastada del projecte: LAMB \'es un ecosistema en \'us actiu a l'ICE--UPC, amb professors i estudiants reals, i la manca d'un sistema d'avaluaci\'o assistida flexible i adaptatiu \'es una limitaci\'o operativa concreta identificada per l'equip.

Els col·lectius que es beneficiaran directament de la soluci\'o s\'on els professors de les institucions que usin LAMB (avaluacions m\'es r\`apides i estructurades amb evid\`encies concretes) i els estudiants (feedback m\'es detallat i accionable). De manera indirecta, qualsevol instituci\'o educativa que adopti LAMB en el futur podr\`a utilitzar Evaluaitor-Lamb sense cost addicional.

Evaluaitor-Lamb no substitueix la figura del professor, sin\'o que l'assisteix: el sistema proporciona una avaluaci\'o preliminar amb evid\`encies i recomanacions, per\`o la decisi\'o final i el feedback qualitatiu continuen recaient en el docent. Aix\`o minimitza el risc de perjudicar la qualitat educativa. A m\'es, la privacitat de les dades acad\`emiques es planteja de manera \`etica i responsable: el disseny prioritza l'execuci\'o local-first amb models Open Source per garantir que la propietat intel·lectual dels estudiants i les seves dades personals mai es cedeixin a prove\"idors cloud externs. Aquest enfocament socialment comprom\`es protegeix la sobirania de les dades de la universitat p\'ublica enfront de les grans corporacions tecnol\`ogiques estrangeres.

\chapter{Prompts del Sistema (System Prompts)}
\label{annex:prompts}
L'arquitectura meta-ag\`entica d'aquest projecte dep\`en en gran mesura de la instrucci\'o inicial donada als models de llenguatge. Aquest annex detalla un dels \textit{prompts} m\'es cr\'itics del sistema: el prompt per generar un \textit{workflow}.

\section{Workflow generator}
\begin{lstlisting}[language=markdown]
# Workflow Generator Prompt

Eres un experto en Ingeniería del Software y evaluación académica. Tu tarea es **generar un workflow JSON** que defina los pasos necesarios para evaluar un documento académico contra una rúbrica dada.

## Catálogo de Tools Disponibles

--{CATALOGO}--

## Rúbrica a Evaluar

--{RUBRICA}--

## Documento de Entrada (Contexto)

--{DOCUMENTO}--

## Instrucciones

1. **Analiza la rúbrica**: Identifica los criterios de evaluación y sus pesos.
2. **Selecciona las tools necesarias**: Elige SOLO las tools estrictamente necesarias para los criterios de la rúbrica.
3. **NO uses `full: true` en `criterion_evaluator`**: A menos que la rúbrica requiera explícitamente análisis profundo de objetivos, requisitos o casos de uso. Para rúbricas de memoria técnica o diagramas, usa solo la evaluación directa.
4. **Define el orden de ejecución**: Las tools de extracción deben ejecutarse antes que las de análisis, y las de análisis antes que las de evaluación.
5. **Conecta los outputs con los inputs**: Usa referencias `${step_id.result.key}` para pasar datos entre steps.
6. **Configura el manejo de errores**: Usa `on_error: "abort"` para steps críticos, `on_error: "skip"` para steps opcionales.
7. **Genera un JSON válido** que cumpla con el siguiente esquema:

```json
{
  "version": "1.0",
  "name": "workflow_<nombre>",
  "description": "<descripción breve>",
  "metadata": {
    "rubric_id": "<id de la rúbrica>",
    "created_at": "<fecha actual ISO8601>",
    "version": "1.0"
  },
  "variables": {
    "input_docx": "path/to/document.docx",
    "input_rubric": "path/to/rubric.yaml",
    "output_dir": "path/to/output"
  },
  "steps": [
    {
      "id": "step_<nombre>",
      "tool": "<nombre de la tool>",
      "description": "<qué hace este paso>",
      "params": { ... },
      "output": { ... },
      "on_error": "abort|skip|retry"
    }
  ]
}
```

## Reglas

- **NUNCA inventes tools** que no estén en el catálogo.
- **Cada step debe tener un `id` único** con el formato `step_<nombre>`.
- **Las referencias a variables** usan `${variable_name}` para variables iniciales y `${step_id.result.key}` para outputs de steps anteriores.
- **El workflow debe ser MINIMALISTA**: no incluyas steps innecesarios. Si la rúbrica solo pide evaluar el documento, usa directamente `criterion_evaluator` con `full: false`.
- **Si la rúbrica tiene criterios de trazabilidad** (matrices obj-req), incluye `detect_orphans`.
- **Si la rúbrica tiene criterios de objetivos**, incluye `evaluate_smart`.
- **Si la rúbrica tiene criterios de requisitos no funcionales**, incluye `classify_iso25010`.
- **Si la rúbrica evalúa diagramas o imágenes** (ej. "Diagrama de clases", "Modelo C4"), incluye `describe_diagrams` después de la extracción del documento para que el evaluador pueda leer la descripción. **IMPORTANTE**: Cuando uses `describe_diagrams`, especifica el parámetro `"model": "qwen3-vl-32b"` (NO uses `qwen-vl-max` ni otros modelos de visión, ya que no están disponibles).
- **EVITA `full: true`** a menos que sea estrictamente necesario.
- **IMPORTANTE**: En la sección `variables`, usa paths genéricos (ej. `path/to/document.docx`) en lugar de paths reales del documento de ejemplo. El workflow debe ser reutilizable para cualquier documento.

## Formato de Respuesta

Responde SOLO con el JSON del workflow, sin texto adicional.
\end{lstlisting}
